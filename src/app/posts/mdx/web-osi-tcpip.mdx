## 학습을 다시 시작하며

최근 네트워크 기초를 다시 살펴보면서, 예전에 막연히 기억했던 “TCP는 양방향, UDP는 단방향” 같은 개념이 얼마나 부정확했는지 깨달았습니다. 
그동안은 단순히 “디버깅을 잘하는 법”에 집중했을 뿐, 문제가 왜 발생했는지를 **OSI 계층 구조와 연관지어 생각해본 적은 없었습니다.**

이번에 개념을 다시 정리하면서, 과거 웹 개발을 하며 접했던 여러 기술 요소들이 자연스럽게 연결되었습니다. 
예전에는 단순히 코드나 기능으로만 이해했던 부분들이, 이제는 네트워크 구조와 맞물리며 훨씬 더 명확하게 다가왔습니다.  

결국 이번 학습은 ‘외워서 아는 것’이 아니라, **경험을 기반으로 이해하는 진짜 학습**이었다고 느꼈습니다.


## OSI 7계층 핵심 요약

| 계층 | 이름 | 핵심 역할 | 대표 프로토콜 | 대표 장비 |
|------|------|------------|----------------|-------------|
| 7 | 응용 (Application) | 사용자 서비스 제공 | HTTP, DNS, SMTP | PC, 서버 |
| 6 | 표현 (Presentation) | 인코딩·압축·암호화 | TLS, SSL, UTF-8 | – |
| 5 | 세션 (Session) | 연결 유지·복구 | RTSP, OAuth2 | – |
| 4 | 전송 (Transport) | 포트·신뢰성 관리 | TCP, UDP | L4 스위치 |
| 3 | 네트워크 (Network) | IP 라우팅, 주소 지정 | IP, ICMP, OSPF | 라우터 |
| 2 | 데이터링크 (Data Link) | 프레임 전송, MAC 주소 관리 | Ethernet, VLAN | 스위치 |
| 1 | 물리 (Physical) | 전기/광 신호 전송 | RS-232, Wi-Fi | 케이블, NIC |

> OSI 7계층은 “개념 지도”, TCP/IP 4계층은 그 지도를 따라 지어진 “실제 도로망”입니다.


## TCP/IP 4계층과 OSI 대응

| TCP/IP 계층 | 역할 | OSI 대응 | 주요 프로토콜 |
|--------------|------|------------|----------------|
| 응용(Application) | 사용자 서비스 | 5~7계층 통합 | HTTP, TLS, DNS |
| 전송(Transport) | 신뢰성, 속도 제어 | 4계층 | TCP, UDP, QUIC |
| 인터넷(Internet) | 라우팅, 주소 관리 | 3계층 | IP, ICMP, OSPF |
| 네트워크 접근(Network Access) | 프레임, 신호 전송 | 1~2계층 | Ethernet, Wi-Fi, VLAN |


## 웹 개발과 OSI 7계층의 만남

웹 디버깅 중 Chrome DevTools의 **Network 탭**을 보면, 요청과 응답이 어떤 경로를 거치는지 상세히 확인할 수 있습니다. 
이때 발생하는 대부분의 문제는 **각 계층별 원인**으로 분류할 수 있습니다.

| 증상 | 확인 위치 | 해당 계층 | 원인 분석 |
|------|------------|-------------|-------------|
| 404 Not Found | Network Status Code | 7 (응용) | 엔드포인트 라우팅 오류 |
| 500 Internal Server Error | Response Body | 7 (응용) | 서버 내부 예외 발생 |
| TTFB 지연 | Timing 탭 (Waiting) | 4 (전송) | TCP 재전송, 혼잡 제어 |
| ERR_CERT_INVALID | 브라우저 경고창 | 6 (표현) | TLS 인증서 만료 |
| ERR_NAME_NOT_RESOLVED | 요청 미출력 | 3 (네트워크) | DNS 해석 실패 |
| Wi-Fi 끊김 | OS 네트워크 상태 | 1 (물리) | 물리적 연결 문제 |

> 단순한 “API 호출 실패”도, 어느 계층의 문제인지 구분할 수 있으면 원인 파악 속도가 달라집니다.


## 실무 적용 사례

### 1️⃣ API 연동 오류
- **CORS 정책 오류 (7계층)** → 서버 CORS 설정 추가
- **응답 지연 (4계층)** → TCP 재전송 또는 서버 응답 지연
- **인증 토큰 만료 (5계층)** → JWT 갱신 로직 필요
- **데이터 포맷 오류 (6계층)** → Content-Type 불일치 (JSON vs HTML)

### 2️⃣ 로그인 흐름 분석
HTTPS 기반 로그인 요청에서:
- 7: `/auth/login` 요청/응답  
- 6: TLS 암호화  
- 5: JWT 세션 유지  
- 4: TCP 443 연결  
- 3: DNS 라우팅  
→ 어느 단계에서 실패했는지 파악 가능

### 3️⃣ 이미지 업로드 & S3 Presigned URL
- 7: URL 만료(403) → 클라이언트 재요청 필요  
- 6: MIME 타입 불일치  
- 4: 대용량 TCP 단편화  
- 1~2: Wi-Fi 신호 불안정 시 재전송 발생  


## 보안 계층별 위협과 대응

| 계층 | 위협 | 대응 기술 |
|------|------|------------|
| 7 | SQL Injection, XSS, CSRF | WAF, 입력값 검증 |
| 6 | TLS 취약, 중간자 공격 | HTTPS, 인증서 검증 |
| 5 | 세션 하이재킹 | Secure 쿠키, 세션 만료 정책 |
| 4 | SYN Flood, 포트 스캔 | 방화벽, IDS/IPS |
| 3 | DDoS, IP 스푸핑 | ACL, VPN, Anti-DDoS |
| 2 | ARP 스푸핑 | DAI, VLAN 분리 |
| 1 | 물리적 도청, 단자 접근 | 출입통제, 케이블 차폐 |


## 개발자가 OSI 7계층을 알아야 하는 이유

| 관점 | 알아야 할 핵심 | 실제 적용 예시 |
|------|----------------|----------------|
| 프론트엔드 | HTTPS, 쿠키 정책, CORS | CORS 오류, 쿠키 SameSite |
| 백엔드 | 세션, JWT, 포트 관리 | API 인증 실패, 포트 충돌 |
| 인프라 | VLAN, 방화벽, NAT | 사내망 통신 불가, DNS 오류 |
| AI·API | HTTPS 인증, JSON 파싱 | STT API 호출 실패, 응답 지연 |

> OSI 7계층은 단순한 네트워크 이론이 아니라, **실무 문제 해결을 위한 사고의 지도(Framework)** 입니다.


## 마무리하며

예전에는 용어를 외우는 데 그쳤다면, 이제는 **각 계층이 실제 서비스 속에서 어떻게 작동하는지**를 직접 체감하며 이해하게 되었습니다. 
자료를 정리하면서 가장 크게 느낀 점은, 결국 문제 해결의 핵심은 ‘계층’을 올바르게 이해하는 데 있다는 것입니다. 
앞으로는 단순히 로그를 확인하는 데서 멈추지 않고, 각 계층의 흐름을 따라가며 문제를 바라보는 습관을 유지하려 합니다.
